# Checkpoint 持久化设计

## 1. 底层实现

checkpoint 当前使用：

- 本地 JSON 文件

作为持久化载体。

核心实现位于：

- `store.py`


## 2. 存储方式

每个 checkpoint 会生成一个独立 JSON 文件。

同时系统还维护：

- 内存缓存 `_active_checkpoints`
- `temp/` 目录下的任务临时目录


## 3. 为什么采用文件存储

当前项目是本地工具，文件存储的优点很明显：

- 部署简单
- 便于直接查看和调试
- 对单机任务型应用足够实用

代价是：

- 并发模型有限
- 不适合多进程共享
- 需要自己处理原子写入和缓存一致性


## 4. 原子写入策略

`CheckpointStore.save_checkpoint(...)` 当前采用：

- 写入 `*.tmp`
- 再移动覆盖正式文件

这样做是为了降低写一半中断时留下损坏文件的概率。


## 5. 缓存策略

store 会把已加载的 checkpoint 放入：

- `_active_checkpoints`

这样可以减少重复读文件的成本。

但这也意味着当前设计更偏向：

- 单进程内使用

而不是跨进程共享状态。


## 6. LLM 响应序列化

LLM 原始响应对象通常不能直接写入 JSON，因此：

- `llm_state.py`

会先把响应序列化为 JSON 兼容结构，再保存到 checkpoint 中。

这保证了恢复时至少能看到：

- message 内容
- tool call 结构
- finish reason 等信息


## 7. 临时目录

每个 checkpoint 会拥有独立的临时目录：

- `checkpoints/temp/<checkpoint_id>/`

它用于承接：

- 中间结果
- 临时文件
- 切片输出等辅助数据


## 8. 设计评价

当前持久化方案非常适合本项目当前阶段：

1. 易调试
2. 易理解
3. 部署成本低

但如果未来出现：

- 多进程执行
- 更高并发
- 更复杂的查询需求

就需要考虑更强的状态后端。
